# A Compact Flat Format for LLM Tool Calling

Imagine you're at a drive-thru. You roll down the window and the order taker asks what you want. You could say:

> "I would like to order one hamburger, and the hamburger should have lettuce on it, and it should also have tomato, and could you please add cheese, and the bun should be toasted, and I'd like a medium fries, and the drink should be a Coke, and the Coke should be large."

Or you could say:

> "Number 3, combo, large Coke."

Same order. Same food. Way fewer words.

That's what this project is about. **tool-reduce** is a middleware for the Vercel AI SDK that teaches models to order their tool calls like a number 3 combo instead of spelling out the full itemized receipt. The result? Less tokens wasted, faster responses, lower costs.

Let me explain why that matters.

## LLMs Speak In Tokens

First, a quick primer. LLMs (Large Language Models) don't read words like we do. They read **tokens** — chunks of text that are roughly 3-4 characters each. The word "hamburger" might be 3 tokens. A period is 1 token. A space is sometimes its own token.

When you use an LLM in an app, two things cost tokens:

1. **Input tokens** — everything you send to the model (your instructions, conversation history, tool definitions)
2. **Output tokens** — everything the model sends back (its responses, tool calls, etc.)

Both cost money and take time. **Output tokens are especially expensive** because the model has to generate them one at a time, and they can't be cached between requests.

In a [Vercel AI SDK](https://sdk.vercel.ai) app, when you define tools for a model, you're telling it: "Here are things you can do — call functions, fetch data, send emails." The model responds with something like:

```json
{
  "type": "tool_use",
  "name": "getWeather",
  "input": {
    "location": "Austin"
  }
}
```

That works. But look at all the structural noise. Curly braces. Quotation marks. Commas. The word "type". The actual useful information is just: "getWeather, location=Austin". Everything else is JSON scaffolding.

**Here's the question that drove this work:**

> **How much of a model's output budget is spent on the envelope vs. the actual content?**

That question is worth sitting with. Every character the model spends on formatting is a character it can't spend on reasoning. Every closing brace is a fraction of a second added to your agent's response time. When your agent runs 10 tool calls in sequence, those fractions add up.

## What We Built

Anthony and I built **tool-reduce** — a middleware layer for the Vercel AI SDK v6 that replaces the verbose JSON tool-call format with a compact one-line format. Instead of that JSON block above, the model emits:

```
<call>getWeather location="Austin" units=metric</call>
```

The middleware intercepts the model's output, parses those `<call>` tags, and converts them back into the real tool-call objects the SDK expects. To the rest of your code, nothing changed. streamText, generateText, onStepFinish, multi-step execution — everything works the same. Just fewer tokens.

## How Much Do You Actually Save?

We ran two sets of benchmarks. One offline (deterministic, ideal conditions) and one live (against a real model, real API).

### What LLM Tokens Cost

At the time of writing, here's roughly what you're paying per million tokens (prices vary by provider — these are ballpark frontier model rates):

| Token Type | Cost per 1M tokens |
|---|---|
| Input tokens | ~$3 - $15 |
| Output tokens | ~$15 - $75 |

Output tokens are **5x more expensive** than input tokens. Saving 45% on output tokens isn't just about speed — it's about cost.

### Offline Benchmark

We measured a catalog of 9 tools (functions the model can call) and 10 representative tool invocations using a 4-char-per-token estimator (matches [tiktoken](https://github.com/openai/tiktoken), OpenAI's official tokenizer, to within ~10%).

| Tool Call | JSON Format | Compact Format | Savings |
|---|---|---|---|
| getWeather(location) | 23 tokens | 11 tokens | 2.09× |
| getTime(timezone) | 24 tokens | 11 tokens | 2.18× |
| webFetch(url) | 25 tokens | 12 tokens | 2.08× |
| sendEmail(to, subject, body, priority) | 40 tokens | 25 tokens | 1.60× |
| searchProducts(query, max, inStock) | 36 tokens | 23 tokens | 1.57× |

The first three tools (simple, 1-2 parameters) save roughly **half** the tokens. The last two (more parameters, some optional) save closer to 40%. The pattern is consistent: the more parameters a tool has, the less proportional savings — but you always save.

Across the full 10-invocation run:

| Metric | Native JSON | Compact | Reduction |
|---|---|---|---|
| Output tokens (total) | 296 | 163 | **44.9%** |
| Round-trip correctness | — | 10/10 | — |

Every single tool call was parsed correctly back to the real tool-call format.

### Live Benchmark (Real Model, Real API)

Offline numbers are one thing. Real inference with a real model tells a different story. We ran each case through [generateText](https://sdk.vercel.ai/docs/ai-sdk-core/generating-text) against **minimax/minimax-m2.5:free** via [OpenRouter](https://openrouter.ai/) — once with native JSON tool calling, once wrapped with tool-reduce.

Configuration: temperature 0 (deterministic), one shot per case, no retries.

| Metric | Native JSON | Compact | Delta |
|---|---|---|---|
| Total output tokens | 803 | 654 | **−18.6%** |
| Total input tokens | 10,898 | 4,308 | **−60.5%** |
| Strict match rate | 8/10 | 9/10 | +1 |
| Soft match rate | 10/10 | 10/10 | — |
| Mean latency | 34.2 s | 31.4 s | — |

The big number here is **input tokens dropping 60%**. Why? Because the tool definitions — the manual you give the model describing what each tool does — shrink dramatically. JSON schema for 9 tools clocks in at ~396 tokens. The compact manual is ~260 tokens. The model also has less structural noise in its own conversation history.

The output reduction (18.6%) is smaller than the offline projection (44.9%) for a simple reason: real models aren't perfect machines. They add extra spaces, quotes, and formatting the idealized compact string doesn't have. Models are trained on JSON, so they naturally drift toward JSON-ish verbosity even in the compact format.

Still, 18.6% fewer output tokens and 60% fewer input tokens is meaningful. In a production agent loop running hundreds or thousands of tool calls per day, that's real cost and latency savings.

### What "Match Rate" Means

When we say "strict match rate," we mean the model called the exact tool with the exact arguments we expected. The misses weren't parsing failures — they were reasonable differences:

- **getWeather**: model returned "Austin, TX" instead of "Austin" (more specific)
- **askDb**: model added a trailing semicolon to SQL (harmless)
- **askDb (compact)**: model put "LIMIT 100" inside the SQL string instead of using the dedicated parameter

Under "soft matching" — where we normalize whitespace, strip trailing punctuation, and check for substring containment — both modes hit 10/10. The model got the right answer every time; it just expressed it slightly differently.

## How It Works (The Simple Version)

The Vercel AI SDK v6 has a feature called [LanguageModelV3Middleware](https://sdk.vercel.ai/providers/community/custom-providers/middleware) that lets you intercept and modify data at three points:

1. **transformParams** — Before the request goes to the model. We strip out the JSON tool definitions and replace them with compact one-line signatures in the system prompt. We also rewrite any previous tool-call history into the same compact format, so the model always sees consistent syntax.

2. **wrapGenerate** — After the model responds. We scan the text for `<call>...</call>` tags and convert them back into real tool-call objects.

3. **wrapStream** — During streaming. A state machine watches for partial `<call>` and `</call>` tags that might be split across chunks (since streaming sends text in pieces), and assembles them correctly.

The middleware supports three encoding styles, chosen automatically based on each tool's [Zod schema](https://zod.dev) (Zod is how the AI SDK defines tool shapes):

- **shell** (default): `<call>getWeather location="Austin" units=metric</call>` — best for flat records of simple values
- **csv**: `<call>getWeather "Austin", metric</call>` — best when all parameters are required
- **json**: `<call>getWeather {"location":"Austin"}</call>` — falls back for tools with nested objects, arrays, or complex types

You don't configure per-tool. The middleware inspects each tool's schema and picks the best encoding automatically.

## This Isn't a New Idea

Making protocols more efficient is a well-established pattern:

- **HTTP/2 HPACK** compresses HTTP headers by replacing repeated key-value pairs with a lookup table. Same concept: stop sending the same structural overhead over and over.

- **Protocol Buffers** (Google's binary format) replaces named fields with numbered field IDs. The JSON tool protocol is basically the equivalent of sending XML for every API call — human-readable but wasteful.

- **Prompt caching** (offered by [Anthropic](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching) and [OpenAI](https://platform.openai.com/docs/guides/prompt-caching)) reuses processed input tokens across requests. The compact format's smaller footprint compounds here — smaller cache entries, cheaper cache misses.

- **Speculative decoding** generates multiple candidate tokens in parallel and keeps the first valid one. It reduces *latency* but doesn't reduce the number of tokens generated. The compact format reduces both.

The difference: all of the above optimize the *transport* — how data moves between systems. tool-reduce optimizes the *generation* — the model itself produces fewer tokens from the start.

## Where It Falls Short

**First-call quality dip.** Frontier models (GPT-5, Claude Sonnet 4.5) are heavily trained on the JSON tool protocol. They follow the compact format from a system prompt, but the very first call in a session might be less accurate. Once the format is in context, it stabilizes.

**Complex tool schemas lose savings.** Tools with nested objects, arrays, or union types can't use the shell/csv syntax — they fall back to JSON inside the `<call>` tag. If your entire toolset looks like `[{user: {name: string, address: {street, city, zip}}}]`, you'll barely save anything.

**No structured output enforcement.** The middleware can't force the model to use a specific output schema (the Vercel middleware interface doesn't expose that). If you need guaranteed structured outputs, use the native JSON format for those specific calls.

**Streaming edge cases.** Splitting `<call>` and `</call>` across chunks during streaming is tricky. The test suite covers 36 edge cases, but it's the most fragile part of the implementation.

## What We Still Don't Know (And Want to Find Out)

- **Multi-step compounding.** We only measured single tool calls. A 10-step agent loop should multiply the per-step savings. But the model might behave differently when it sees its own compact output in context.

- **Frontier model variance.** We tested one model (minimax/m2.5). How do GPT-5 and Sonnet 4.5 handle the compact format? They're trained more heavily on JSON tool calling, so the first-call quality dip might be worse — or they might adapt faster.

- **Better evaluation.** Our strict deep-equality check is too harsh. A small judge model that scores semantic match ("Austin" ≈ "Austin, TX") would give a more accurate picture.

- **Schema flattening.** We currently fall back to JSON for nested schemas. Dotted-key flattening (`address.city`, `address.zip`) and array-of-primitive encoding would extend savings to more real-world tool sets.
